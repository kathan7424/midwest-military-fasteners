Part 1 of 3

The one cron job

Every other email in this build fires off a real event. This is the single exception — a recurring check with no user action behind it.

1
HOOK: mmf_tax_exemption_daily_reminders

Registered once, fires every day

wp_schedule_event( time() + HOUR_IN_SECONDS, 'daily', 'mmf_tax_exemption_daily_reminders' ) — set up on init, guarded so it only ever gets scheduled once.

2
WHAT IT CHECKS

Every approved tax exemption certificate

Loops every customer with mmf_tax_exemption_status = approved, works out days-left-until-expiry for each one.

3
WHAT IT SENDS

Whichever of the 3 reminder emails is due

≤3 days left → urgent · ≤30 days left → expiring soon · already past → expired. Each has its own de-dupe flag so it only ever sends once per expiry date.

Not a real system cron WP-Cron only fires on a page load. A quiet dev site with no visitors can leave this "due" for hours. Force it with wp cron event run mmf_tax_exemption_daily_reminders.

Part 2 of 3

Product Certificate flow

Free per-product certification (MFR C of C, material certs, test reports). One email, zero cron — purely a reaction to an order status change.

1
SETUP

Admin sets a certificate on the product

_certificate_file_url custom field on the physical product — no separate $0 product needed.

2
CHECKOUT

Customer opts in for that line item

Stamped as _mmf_cert_opted_in = '1' on the order line item, permanently, at order-creation time.

3
FULFILLMENT

Order reaches Completed or Shipped

Admin manually changes the status, OR the Shippo webhook auto-completes it on a DELIVERED event — identical result either way.

4
EMAIL FIRES Custom

"Your product certificates are ready"

Only if: opted in AND a cert file was stamped AND not already sent. Silent no-op otherwise.

WC Native WC's own "Completed" order email fires in the same request — two emails, not a duplicate bug.

5
AFTER

Customer re-downloads anytime

My Account → Certifications tab reads the exact same stamped line-item data the email used.

Worked example

Order #4402 — 2× MMF-TEST-002 (has a cert file), opted in.

Admin marks it Completed on July 22.

→ Certificate email fires with a download link. → WC's own Completed email fires too. → _mmf_cert_email_sent = '1' is set — re-marking it Completed later never re-sends.

Part 3 of 3

Tax Exemption flow

Sales-tax exemption for qualifying business customers. This one has an ongoing time dimension — certificates expire — which is exactly why it's the flow with a cron.

1
UPLOAD

Customer uploads a certificate + expiry date

At registration (Gravity Forms) or later from My Account. Status becomes "pending."

2
EMAIL FIRES Custom

Admin review request, ~10 seconds later

Queued so it never slows the registration response. Signed one-click Approve/Reject links, no login required, expire after 14 days.

3
DECISION

Admin approves or rejects — any of 6 paths

The email buttons, the Tax Certificates dashboard (3 ways), the REST API, or Users → Edit — all 6 behave identically.

4
EMAIL FIRES Custom

Approved/rejected notice — instant

If approved: Net 30 is silently enabled at the same time, and tax exemption goes live immediately at checkout.

5
EMAIL FIRES Custom Cron

Expiry reminders — cron every day, PLUS instantly at approval

The daily cron checks every approved cert going forward. But if the approval in step 3 already lands inside the 3-day (or expired) window, the matching reminder fires immediately, same request — see the example below.

Worked example — why the instant check matters, not just the cron

Customer uploads a certificate on July 19 with an expiry of July 22 (today).

Admin doesn't review it until July 22 — the expiry date itself — then clicks Approve.

→ Approval email fires. → days_left = 0 falls into the "≤3 days" bucket → the urgent 3-day reminder fires in that same click, not tomorrow.

Without the instant check, the customer gets approval confirmed but zero warning it expires today — they'd only find out when tomorrow's cron runs, by which point it's already the expired email, not the 3-day one.

These two flows never share state Product Certificate opt-in (_mmf_cert_opted_in, on an order line item) and Tax Exemption status (mmf_tax_exemption_status, on the customer's account) are completely separate records. Approving one has zero effect on the other.
