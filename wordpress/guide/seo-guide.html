1
ALREADY WORKS — NO SETUP NEEDED

Per-page SEO stays 100% inside Yoast

Nothing changes about how you edit SEO day to day. Open any Page, Post, or Product in wp-admin, scroll to the Yoast box at the bottom, and fill in the SEO title, meta description, and social image exactly like always.

The frontend reads this automatically — Yoast attaches a yoast_head_json field to WordPress's own REST API for every page/post/product, and Next.js pulls it at build/request time.

Pages covered right now Home page, every WP Page (About, Quality, Contact, etc.), and every individual Product Detail page — each gets its own Yoast-driven title, description, canonical URL, and social preview.
2
WP ADMIN → SETTINGS → SEO & ANALYTICS

Fill in GTM / GA4 — the one screen that matters

This is a new settings page built specifically for this. Three fields:

FieldFormatWhat it's for
GTM Container IDGTM-XXXXXXXLoads the full Google Tag Manager snippet on every page
GA4 Measurement IDG-XXXXXXXXXXOnly fill this in if you're NOT routing GA4 through GTM — using both double-fires GA4
Default Social Share ImageMedia Library URLFallback image for pages Yoast has nothing set for (see step 4)
Validation is strict on purpose Anything that doesn't match the exact GTM-XXXXXXX / G-XXXXXXXXXX pattern is silently rejected on save — a typo can never leak into a live script tag on the storefront.
3
AUTOMATIC — NOTHING TO DEPLOY

GTM/GA4 scripts appear the moment you save

Those two IDs travel through the existing site-settings API — the same one already fetched on every single page for the header/footer/logo. No extra network round trip, no separate endpoint to maintain.

On the Next.js side, the scripts use next/script (Next's own documented pattern for GTM — not a raw <script> tag), and they're fully conditional:

  • No GTM ID saved → no GTM script renders at all, anywhere
  • No GA4 ID saved → no GA4 script renders
  • Fill either one in → it appears on every page on the next request, automatically
4
FALLBACK ONLY — YOAST ALWAYS WINS FIRST

Social share image — global fallback, per-page override

Priority order, every time a page is shared on Facebook/LinkedIn/etc.:

  1. Yoast's own image for that specific page (set in the Yoast box, or auto-picked from the featured image)
  2. If Yoast truly has nothing → the Default Social Share Image from step 2
  3. If neither exists → no image tag renders (never a broken image link)

This means you almost never need to touch the default — it only ever shows up for pages nobody has bothered to set a specific image for yet.

5
ALREADY WORKING

Favicon

Set once under WP Admin → Settings → General → Site Icon — this is the real favicon shown in browser tabs, bookmarks, and mobile home-screen icons, and it's already configured and confirmed live.

About the "GET /favicon.ico 404" dev log line Some browsers additionally probe /favicon.ico on your own domain as a legacy habit, separate from the real Site Icon (which lives on the WP domain and is correctly wired via the page's own <link rel="icon"> tag). A code-level fallback icon now exists at /icon.svg for whenever the Site Icon isn't set — but that one specific log line is cosmetic dev-server noise, not something a visitor ever sees.
